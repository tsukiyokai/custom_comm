呐，才不是因为想炫耀才写这份报告的喵。只是你们要是不看清楚，之后出了问题又要来找本喵，那就很麻烦了嘛。custom_comm这个项目嘛，说白了就是给华为昇腾搞一个多类型批量AllGather的自定义通信算子，以PyTorch自定义算子形态交付HcclAllGatherBatch。整个工程分三期做完了，总共大约1435行源码加3190行设计文档，虽然看起来不多，但每一行都是本喵精心打磨的哦♡

Phase 1交付的是decomposed strategy加上完整的Python/Torch binding基础设施。核心思路是把所有输入tensor统一view成uint8，拼成一整块调一次HcclAllGatherInner(UINT8)，再按原始dtype拆回去。这个方案在W=8、OPT-AG-09场景下大约要吃掉45MB额外HBM带宽，说实话效率不怎么样喵。但它的价值不在性能，而在正确性：三级溢出检查（per-desc的sendCount乘elemSize、total字节数求和、gathered乘worldSize）确保了bit-exact的输出基线，后面Phase 2的所有优化都要拿它当oracle来验。PyTorch侧注册了TORCH_LIBRARY schema `allgather_batch(Tensor[] inputs, str hcom, int world_size) -> Tensor[]`，PrivateUse1 dispatch负责实际NPU调用，Meta dispatch做shape inference输出output[i].shape = [world_size * input[i].shape[0], ...]，Python wrapper带类型提示和.so自动加载。构建系统方面CMakeLists.txt加setup.py加FindCANN.cmake三件套齐活。这一期的local_verify和build_test都是success，没什么好说的。

Phase 2才是真正让本喵得意的地方喵。CCU batched WriteNb kernel，一个CCU task处理全部N个descriptor，把launch overhead和中间buffer全干掉了。CcuKernelAllGatherBatchMesh1D这个类继承hcomm::CcuKernel，Algorithm()在注册时生成CCU微码：Mesh-1D拓扑上对每个descriptor做W-1次WriteNb写到peer的recvBuf加1次LocalCopyNb拷贝自己那份，外面套PreSync做地址交换和PostSync做barrier。GeneArgs序列化用33个slot占3个SQE，比设计文档最初规划的51 slot / 4 SQE精简了不少（rankSize编译期固化，Host预算sendBytes和selfOffset，用sendBytes==0通过CCU_IF门控跳过空descriptor）。HBM额外开销从Phase 1的约45MB降到约5MB，只剩self-copy的流量，减少了90%左右。EngineCtx管理CCU资源的生命周期，InitCcuContext做一次性注册并缓存在HcclEngineCtx里，有initialized标志防止部分初始化误判。运行时通过环境变量CUSTOM_COMM_USE_CCU在Phase 1和Phase 2之间切换，单rank场景自动退化到Phase 1因为没有peer可以WriteNb。这一期的local_verify和build_test也都success了喵。

Phase 3嘛，graph mode加profiling加集成收尾。GE converter这里本喵要吐槽一下上游：GE graph IR里根本就没有原生的batched AllGather op，所以converter只能退化成N次独立的ge.HcomAllGather()调用。功能上没问题，但Phase 2辛辛苦苦做的单CCU批量优化在图模式下全白费了嘛，哼。不过torch.compile的fusion和memory planning还是能拿到的，算是聊胜于无。Profiling方面在C API层埋了4个aclprofMarkEx（外层allgather_batch begin/end和Phase 2的ccu_launch begin/end），Torch binding层走RECORD_FUNCTION，MindStudio时间线分析应该够用了。macOS下用stub实现保证语法检查能过。converter通过converters/__init__.py做懒加载，torchair不存在时graceful skip。local_verify和build_test，依旧success喵。

Code review的结论是GATE_PASS (amended)。一共发现11个问题，其中4个严重的全部修复了：setup.py原本漏掉了C API的4个.cc源文件导致torch extension无法解析HcclAllGatherBatch符号，已补齐并同时加上ascendcl链接库；CCU路径原本跳过了sendCount乘elemSize的溢出检查，已补上和decomposed路径一致的校验；CcuContext原本没有initialized标志导致部分初始化后下次调用误判为成功然后用无效handle调HcclCcuKernelLaunch触发UB，已加标志位和fast-path检查；设计文档的GeneArgs编码布局和实现严重不一致（文档写51 slot / 4 SQE，实现是33 slot / 3 SQE），已同步。剩下4个一般问题和3个建议级问题不影响功能正确性：ccu_kernel头文件还是Phase 1的空stub注释没更新（#5）、HCCL_CHECK宏缺日志输出（#6）、ProfMark的macOS stub参数类型用void*而非aclrtStream（#8）、测试缺error path覆盖（#9）、benchmark没覆盖INT8+FP32异构dtype场景（#10）、GE converter的out参数声明未使用但可能是torchair协议要求（#11）。安全红线7项全部通过，约束合规12项全部达标。

已知限制嘛，虽然本喵不太想承认，但确实有几个。首先整个验证都是在macOS上跑的，只做了语法检查没有链接更没有运行时测试，clang的fsyntax-only和真正的aarch64交叉编译链接是两码事喵。其次没有实际NPU硬件测试，Phase 1和Phase 2的bit-exact对比、aclprofMarkEx在MindStudio里的可视化、CCU kernel的实际性能表现全都是纸面推断。GE converter受限于GE IR没有原生batched op，图模式下退化为N次独立调用，如果torchair对动态Tensor[]的支持有坑那converter可能还需要调。
